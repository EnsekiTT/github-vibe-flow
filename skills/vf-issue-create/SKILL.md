# Vibe Flow Issue Create

## 概要

`/vf-design` で作成された設計ドキュメントからGitHub Issueを生成する。

**開始時の宣言:** 「vf-issue-create スキルを使って、設計ドキュメントからGitHub Issueを生成します。」

## プロセス

### 1. 設計ドキュメントの読み込み

引数として設計ドキュメントのパスを受け取るか、対話的に確認する。

```bash
# ファイルの存在確認
ls <design-doc-path>
```

ドキュメントを読み込み、以下を抽出する:
- Issue分解（タイトル、説明、受入基準）
- 実行記法（仮ID）

### 2. GitHub Issueの作成

分解された各Issueに対して、MCP経由でGitHub Issueを作成する:

**タイトル:** Issue分解のタイトルから取得

**本文フォーマット:**
```markdown
## 説明

[設計ドキュメントからの説明]

## 受入基準

- [ ] [基準1]
- [ ] [基準2]

## 設計ドキュメント

[設計ドキュメントへのリンクまたはパス]

---
*Generated by github-vibe-flow*
```

**ラベル:** `vibe-flow` ラベルを付与（存在しない場合は作成）

### 3. 実行記法の更新

仮IDを実際のGitHub Issue番号にマッピングし、設計ドキュメントを更新する:

- 更新前: `[1, (2, 3), 4]`
- 更新後: `[#12, (#13, #14), #15]`

更新した記法を設計ドキュメントに書き戻し、コミットする。

### 4. 人間への確認

作成されたIssueと更新された実行記法を表示する:

```
作成されたGitHub Issue（4件）:
  #12: [タイトル1]
  #13: [タイトル2]
  #14: [タイトル3]
  #15: [タイトル4]

実行順序: [#12, (#13, #14), #15]

この内容で問題ないですか？
```

### 5. ハンドオフ

- `/vf-flow` から呼び出された場合: 人間の確認後、実行記法を持って `/vf-execute` に進む
- 単体で呼び出された場合: 完了

## エラーハンドリング

- MCPが未設定の場合: GitHub MCP Serverの設定方法を案内する
- ラベル作成が失敗した場合: ラベルなしで続行し、警告を表示する
- Issue作成が失敗した場合: 成功・失敗したものをそれぞれ報告する

## 連携

**呼び出し元:**
- vf-design（ハンドオフ）— `/vf-flow` から呼び出された場合
- vf-flow — 2番目のステップとして

**呼び出し先:**
- GitHub MCP Server — Issue作成、ラベル管理
