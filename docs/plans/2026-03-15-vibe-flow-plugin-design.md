# github-vibe-flow Plugin Design

## Overview

GitHub Issue駆動でCoding Agentと人間が協働するためのClaude Code Plugin。
人間は「起点（きっかけ）」と「関門（レビュー承認）」に集中し、実行はAgentが自律的に担う。

## Dependencies

- Claude Code Plugin機構
- GitHub MCP Server（Issue・PR・コメント等のGitHub操作）
- tmux（作業環境前提）
- git worktree（並列実行時のAgent作業領域隔離）
- superpowers plugin（brainstorming等の既存スキル活用）

## Skill一覧

| Skill | 役割 |
|---|---|
| `/vf-flow` | メタフロー。設計→Issue作成→実行→監視→クリーンアップを一気通貫で実行 |
| `/vf-design` | brainstormingスキルを呼び出し、設計ドキュメント作成 |
| `/vf-issue-create` | 設計ドキュメントからGitHub Issueを生成 |
| `/vf-execute` | 実行記法に基づきAgentがIssueに着手（worktree + tmux） |
| `/vf-monitor` | 常駐監視Agent。PRコメント検知→自動レビュー対応 |
| `/vf-merge` | フロー完了後のクリーンアップ（worktree・ブランチ・設計ドキュメント削除、ローカル同期） |

## E2Eフロー（`/vf-flow` が統括）

1. `/vf-design` → 設計ドキュメント作成
2. `/vf-issue-create` → GitHub Issue生成
3. `/vf-execute` → worktree + tmuxでAgent実行
4. Agent作業完了 → PR作成 → セルフレビュー＆改善ループ → 人間レビューReady
5. `/vf-monitor` → 人間のレビューコメント検知 → Agent自動対応
6. 人間Approve → マージ
7. `/vf-merge` → クリーンアップ

### 人間の関与ポイント

| タイミング | 人間のアクション |
|---|---|
| 設計完了後 | brainstorming内で対話的に確認済み |
| Issue生成後 | 実行記法・Issue内容の確認 |
| 実行中 | 介入不要（Agentが自律実行） |
| レビューReady | PRのレビュー・Approve |
| 全PRマージ後 | クリーンアップの承認 |

## `/vf-design` — 設計スキル

### 目的

人間のアイデアを設計ドキュメントに落とし込む。superpowersのbrainstormingスキルをラップする。

### フロー

1. 人間がきっかけ（テーマ・課題）を入力
2. 内部でsuperpowers:brainstormingスキルを呼び出し、対話的に設計を詰める
3. 完成した設計ドキュメントを `docs/plans/YYYY-MM-DD-<topic>-design.md` に保存
4. 設計ドキュメントのパスを返す（後続の `/vf-issue-create` への入力となる）

### 設計ドキュメントに含める追加項目（brainstorming標準に加えて）

- **Issue分割案** — 設計を実装可能なIssue単位に分割した一覧
- **実行記法** — Issue間の依存関係を `[1, (2, 3), 4]` 記法で記述（この時点ではまだGitHub Issue番号がないため仮番号）
- **受け入れ基準** — 各Issueの完了条件

### 補足

- `/vf-flow` 経由で呼ばれた場合、完了後自動的に `/vf-issue-create` に遷移
- 単体で呼ばれた場合は設計ドキュメント保存で完了

## `/vf-issue-create` — Issue生成スキル

### 目的

設計ドキュメントからGitHub Issueを自動生成する。

### フロー

1. 設計ドキュメントのパスを入力として受け取る（引数 or 対話で指定）
2. 設計ドキュメント内のIssue分割案を読み取る
3. 各IssueについてGitHub MCP経由で作成:
   - タイトル
   - 本文（実装内容・受け入れ基準・関連する設計ドキュメントへのリンク）
   - ラベル（`vibe-flow` で識別）
4. 作成されたGitHub Issue番号を設計ドキュメントの仮番号にマッピング
5. 実行記法を実Issue番号で更新し、設計ドキュメントに書き戻す
   - 例: `[1, (2, 3), 4]` → `[#12, (#13, #14), #15]`
6. 更新された実行記法を表示し、人間に確認を求める

### 補足

- `/vf-flow` 経由の場合、人間の確認後に `/vf-execute` に遷移
- 単体の場合はIssue作成完了で終了

## `/vf-execute` — 実行スキル

### 目的

実行記法に基づき、tmux + git worktreeで複数Agentを管理しながらIssueを実行する。

### 入力

- 実行記法（例: `[#12, (#13, #14), #15]`）— 引数で直接指定、または設計ドキュメントのパスから読み取り

### 実行記法

- `[]` — シリアル（順次実行）
- `()` — パラレル（並列実行）
- 例: `[#12, (#13, #14), #15]` → #12完了後に#13と#14を並列実行、両方完了後に#15

### フロー

1. **環境チェック**
   - 現在tmuxセッション内か確認。内ならそのセッションを利用、外なら新規セッション作成して接続
2. **実行記法をパース**
   - 実行グラフ（DAG）を構築
3. **各Issueの実行（DAGに従って）**
   - git worktreeを作成（ブランチ名: `vf/#12-issue-title` 等）
   - tmuxに新しいペインを作成
   - そのペイン内でClaude Codeを起動し、Issueの内容を指示として渡す
   - Agentの指示内容: Issue本文 + 受け入れ基準 + 「完了したらPR作成 → セルフレビュー＆改善ループ → レビューReady判定」までを含む
4. **進捗管理**
   - 各ペインの状態（実行中・PR作成済み・レビューReady）を追跡
   - パラレルグループ内の全Agentが完了したら、次のシリアルステップに進む
5. **全Issue完了後** — `/vf-monitor` の起動を提案

### セルフレビュー＆改善ループ（各Agentが自律的に実行）

1. PR作成
2. PRの差分をセルフレビューし、改善ポイントをPRコメントに記載
3. コメント内容に基づき修正をコミット＆プッシュ
4. 修正内容をPRコメントに記載
5. 人間レビューReadyか判定 → Readyでなければ2に戻る、Readyなら完了

## `/vf-monitor` — 監視スキル

### 目的

専用のtmuxペインで常駐し、PRへの人間のレビューコメントを検知して自動でAgentに対応させる。

### フロー

1. **起動**
   - tmuxに監視用ペインを作成
   - 監視対象のPR一覧を取得（`vibe-flow` ラベルのついたopen PR）
2. **ポーリングループ**
   - 一定間隔（デフォルト30秒、引数で変更可能）でGitHub MCP経由にて監視対象PRの新規コメント・レビューを確認
   - 検知対象:
     - `changes_requested` レビュー
     - 個別のレビューコメント
     - 通常コメント
   - Agentのセルフレビューコメントは除外（自分自身のコメントに反応しないよう）
3. **レビュー対応の起動**
   - 新規コメント検知時、該当PRのworktreeが残っていればそのまま利用、なければ再作成
   - tmuxに新しいペインを作成し、Claude Codeを起動
   - 指示内容: レビューコメントの内容 + 「修正 → PRコメントで報告 → セルフレビュー＆改善ループ → レビューReady判定」
4. **終了条件**
   - 監視対象PRがすべてマージ or クローズされたら自動終了
   - 人間が手動で停止も可能

## `/vf-merge` — クリーンアップスキル

### 目的

フロー完了後のリソースを一括で片付け、ローカル環境をクリーンな状態に戻す。

### フロー

1. **対象の特定**
   - `vibe-flow` ラベルのついたマージ済みPRを一覧取得
   - 関連するworktree・ブランチ・設計ドキュメントを特定
2. **人間に確認**
   - クリーンアップ対象の一覧を表示し、承認を求める
3. **クリーンアップ実行**
   - git worktreeの削除
   - リモート・ローカルブランチの削除
   - 設計ドキュメントの削除（またはarchiveディレクトリへの移動）
   - tmuxの不要なペイン・ウィンドウのクローズ
4. **ローカル環境の同期**
   - デフォルトブランチ（main/master）に切り替え
   - `git pull` で最新状態に更新
5. **結果表示**
   - 削除されたリソースのサマリーを表示

### 補足

- `/vf-flow` 経由の場合、全PRがマージされた後に自動提案
- 単体でも実行可能
- 設計ドキュメントは削除かアーカイブか、人間に選択させる

---

## 中断・再開

- 途中で中断した場合、各スキルを単体で呼び出して任意のステップから再開可能
- `/vf-flow` はあくまで便利なオーケストレーターであり、各スキルの独立性は維持
